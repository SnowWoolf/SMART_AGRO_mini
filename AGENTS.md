# Repository Guidelines

## Язык общения

С пользователем этого репозитория нужно говорить и переписываться только на русском языке. Ответы, пояснения, вопросы, комментарии к выполненной работе и итоговые сообщения должны быть на русском.

## Согласование изменений

Перед внесением любых изменений в файлы нужно описать пользователю планируемые правки, зачем они делаются и ожидаемый результат. Применять изменения к файлам можно только после явного подтверждения пользователя.

После фактического внесения изменений в файлы нужно отдельным сообщением написать: `внесены изменения в файлы: {список}`. В списке указывайте все реально измененные файлы.

## Версионирование

После фактического внесения изменений в файлы обновляйте файл `version` по схеме `2.0.{текущая дата}.{номер}`. Дату записывайте в формате `ДДММГГ`, как в текущем файле `version`.

Если дата в `version` совпадает с фактической текущей датой, увеличивайте `{номер}` на 1. Если дата в `version` старая, меняйте ее на сегодняшнюю и ставьте `{номер}=1`.

Не меняйте `version`, если редактировались только файлы `*.md`, `*.db`, `*.txt`, `.env`, `main.html`, `main2.js`, `update.sh` или `migrate*.py`.

Для каждого файла, в который вносятся изменения, в первой строке нужно указывать новый номер версии, совпадающий с номером, который записывается в `version`. Формат строки версии должен быть безопасным для языка файла: Python, shell и `.env` через `#`, JavaScript через `//`, CSS через `/* ... */`, HTML/Jinja через `<!-- ... -->`. Не добавляйте строку версии в `*.md`, `*.txt`, `*.db`, изображения, иконки, бинарные файлы и сам файл `version`. Если у файла есть обязательная первая строка, например shebang в `*.sh`, оставляйте ее первой, а строку версии размещайте сразу после нее. Для `main.html` и `main2.js` используйте формат `univ-{номер}` вместо `universal`.

## Структура проекта и организация модулей

Это локальное Flask-приложение и набор фоновых сервисов для управления и синхронизации SMART AGRO mini. Проект рассчитан на работу на целевом Linux-устройстве, но в этой среде код может редактироваться с Windows.

- `run.py` запускает веб-сервер разработки на `0.0.0.0:5555`.
- `app_instance.py` создает общий экземпляр Flask-приложения.
- `app/` содержит основной пакет веб-приложения.
- `app/routes.py` содержит маршруты веб-панели, API для фронтенда, работу с камерой, сетевыми настройками, импортом/экспортом БД и системными командами. Файл большой и монолитный; держите изменения точечно и не переносите крупные блоки логики без отдельной задачи.
- `app/models.py` и `app/forms.py` содержат модели базы данных и WTForms-формы.
- `app/templates/` содержит Jinja HTML-шаблоны.
- `app/static/styles/`, `app/static/scripts/` и `app/static/images/` содержат CSS, JavaScript и изображения.
- `sync_module.py` - долгоживущий сервис `agrosmart_sync`: опрашивает Modbus RTU/TCP, синхронизирует значения параметров с БД, выполняет сценарии, управляет растворным узлом, пишет логи и отправляет уведомления.
- `camera_saver.py` - фоновый архиватор RTSP-камер.
- `migrate_*.py` - ручные миграции SQLite. Перед запуском проверяйте имя БД внутри скрипта.
- `base.db` - локальная SQLite-база с рабочими таблицами и данными; не коммитьте сгенерированные изменения базы без явной необходимости.

## Ключевые домены приложения

- Авторизация и роли пользователей: `User`, `flask_login`, проверка администратора в отдельных маршрутах.
- Параметры оборудования: таблица `parameter`, Modbus-настройки, текущие значения, коэффициент `K`.
- Сценарии и циклы: `Scenario` и `ScenarioCycle`, конструктор сценариев, генерация шагов по времени.
- Растворный узел: `MixingParameter`, pH/EC, дозирование A/B/кислоты, перемешивание, калибровка датчиков.
- Посадки и культуры: `Culture`, `Tray`, `Planting`, расчет готовности и массы урожая.
- Камеры: RTSP-захват, архив JPEG, выдача снимков и сборка timelapse.
- Системные настройки устройства: WiFi, GSM-модем, Ethernet, статусы связи, перезапуск сервисов и перезагрузка.

## Команды сборки, тестирования и разработки

Проект работает на Python 3.8. Для локального запуска, проверок и команд разработки используйте именно Python 3.8.

Перед установкой зависимостей используйте виртуальное окружение:

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python run.py
```

В Windows PowerShell активируйте окружение командой `.venv\Scripts\Activate.ps1`.

Полезные команды эксплуатации из `README.md`:

```bash
systemctl status agrosmart_web
systemctl status agrosmart_sync
```

`install.sh` предназначен для первичной настройки целевого устройства через curl-команду из документации.

В продакшене веб-приложение запускается через `waitress-serve --listen=0.0.0.0:5555 app_instance:app`, а синхронизация - через `python3 sync_module.py`.

Для быстрой синтаксической проверки измененных Python-файлов используйте:

```bash
python -m py_compile app/routes.py app/models.py app/forms.py sync_module.py camera_saver.py
```

## Стиль кода и соглашения об именах

Следуйте текущему Python-стилю: отступы 4 пробела, `snake_case` для функций и переменных, `PascalCase` для классов. Держите Flask-маршруты в `app/routes.py`, изменения ORM в `app/models.py`, а валидацию форм в `app/forms.py`.

Шаблоны называйте описательно и в нижнем регистре, как `scenario_constructor.html`. Статические файлы храните по типам внутри `app/static/`.

Для фронтенд-изменений меняйте согласованную тройку файлов: шаблон в `app/templates/`, JS в `app/static/scripts/`, CSS в `app/static/styles/`. Не добавляйте внешние CDN-зависимости без необходимости: интерфейс должен оставаться пригодным для локальной панели устройства.

## Работа с БД и миграциями

- Модели SQLAlchemy объявлены в `app/models.py`, но `db.create_all()` не заменяет миграции существующей SQLite-базы.
- При изменении схемы добавляйте отдельный миграционный скрипт по образцу `migrate_scenario_cycles.py` и явно указывайте, какую БД он меняет.
- Перед импортом или заменой БД проверяйте структуру таблиц. Маршрут импорта БД перезапускает сервисы, поэтому такие изменения влияют на эксплуатацию устройства.
- Не меняйте `base.db` ради локальных экспериментов. Если изменение БД необходимо, отдельно описывайте причину и способ проверки.

## Правила тестирования

Автоматических тестов сейчас нет. Для новых тестов используйте `pytest`, размещайте их в `tests/` и называйте файлы по шаблону `test_routes.py` или `test_models.py`. Покрывайте модели, права доступа к маршрутам, валидацию форм и логику синхронизации. Пока тестов нет, проверяйте приложение вручную через `python run.py` и затронутые страницы в браузере.

Минимальная ручная проверка зависит от зоны изменений:

- Для главной панели: логин, `/`, обновление параметров, переключатели, последние логи.
- Для сценариев: `/scenario_constructor`, сохранение/удаление цикла, генерация записей `Scenario`.
- Для растворного узла: `/mixing_parameters`, сохранение числовых параметров, калибровочные действия без лишней записи в Modbus.
- Для посадок: `/plantings`, посадка, срезка, освобождение и пересчет аналитики.
- Для сетевых настроек: проверяйте только на целевом устройстве или с явными моками, потому что код пишет в `/etc` и вызывает `systemctl`.

## Коммиты и Pull Request

Git-история в этой среде недоступна, поэтому используйте короткие повелительные сообщения коммитов, например `Add scenario cycle migration` или `Fix camera capture timeout`. Один коммит должен закрывать одну логическую задачу.

Pull Request должен содержать краткое описание, затронутые страницы или сервисы, влияние на конфигурацию или базу данных, шаги ручной проверки и скриншоты для видимых UI-изменений.

## Безопасность и конфигурация

Секреты и значения, зависящие от устройства, храните в `.env`; не прописывайте учетные данные или аппаратные адреса в коде. Внимательно проверяйте изменения в `config.py`, `install.sh`, именах сервисов и миграциях базы данных, потому что они влияют на развертывание.

Учитывайте аппаратные и системные интеграции:

- `sync_module.py` работает с Modbus RTU/TCP через `minimalmodbus`; не меняйте масштабирование `K`, типы регистров и режимы чтения/записи без проверки на совместимость с таблицей `parameter`.
- Веб-маршруты могут вызывать `systemctl`, `/usr/sbin/reboot`, `/sbin/ip`, `/usr/sbin/iw`, `wpa_cli` и писать конфиги в `/etc`. Такие изменения должны быть защищены проверкой прав и понятной обработкой ошибок.
- RTSP-URL, токены MAX, параметры камер, сети и устройства должны приходить из `.env` или системных конфигов.
- Не запускайте команды перезагрузки, перезапуска сервисов или миграции БД без явного понимания последствий для устройства.
